# 在线赛鸽赛事报名系统产品开发说明文档

## 1. 项目概述

本项目是一个用于赛鸽赛事在线报名的 Web 系统，主要面向赛鸽俱乐部、鸽会、赛事组织者以及参赛会员使用。

系统需要支持管理员在后台创建赛事、配置不同报名项目，并通过 Excel 导入会员及足环数据。会员通过手机号和密码登录后，只能看到自己名下的赛鸽足环号码，并可以在手机端完成赛事报名。

系统的核心业务不是普通商品购买，而是围绕以下几个对象展开：

```text
会员账号
↓
会员棚号 / 参赛名
↓
足环号码
↓
赛事
↓
报名项目
↓
单羽报名 / 多羽组合报名
↓
实时金额计算
↓
后台统计与导出
```

本系统必须重点解决三个问题：

1. 管理员可以自由配置报名项目，例如单羽 50 元、单羽 100 元、双羽组 200 元、三羽组 300 元、五羽组 500 元等。
2. 会员在手机端可以清晰、快速、准确地为自己的赛鸽选择报名项目，并实时看到金额。
3. 系统部署在小型腾讯云轻量应用服务器上时，仍然能够承受赛事报名高峰期约 300 人同时访问，避免频繁数据库访问，保证报名流程稳定。

---

# 2. 产品目标

## 2.1 后台目标

管理员可以完成以下操作：

1. 创建会员账号。
2. 设置会员手机号、密码、会员棚号、参赛名。
3. 上传 Excel 表格导入足环数据。
4. 创建赛事。
5. 设置赛事报名时间、截止时间和状态。
6. 为赛事创建多个报名项目。
7. 自定义每个报名项目的名称、金额、每注所需羽数。
8. 查看所有会员报名记录。
9. 查看每个赛事、每个项目、每个会员的报名统计。
10. 导出报名明细、财务汇总、足环统计等 Excel 文件。

## 2.2 会员端目标

会员可以完成以下操作：

1. 使用手机号和密码登录。
2. 查看自己的会员棚号和参赛名。
3. 查看自己名下的足环号码。
4. 查看当前可报名赛事。
5. 在单羽组中，通过列表矩阵形式快速勾选报名项目。
6. 在多羽组中，按项目要求选择指定数量的足环组成一组。
7. 实时查看报名数量和总金额。
8. 查看报名明细。
9. 确认并提交报名。
10. 查看自己已提交的报名记录。

## 2.3 性能目标

系统应按照以下性能目标设计：

1. 支持报名高峰期约 300 名会员同时访问。
2. 会员进入报名页面时，尽量只请求一次初始化数据。
3. 会员在页面内勾选、取消、组队、计算金额时，全部在前端本地完成，不频繁请求服务器。
4. 会员只有在进入赛事报名页、提交报名、查看历史报名时才访问后端。
5. 赛事项目配置、会员足环数据等读多写少的数据需要缓存。
6. 报名提交必须使用后端事务，保证数据准确。
7. 后端最终重新校验金额、足环归属和项目规则，不能信任前端金额。
8. 静态资源通过 Nginx 缓存和压缩，减少服务器压力。
9. Excel 导入、导出等耗时任务尽量异步处理，避免影响报名高峰。

# 3. 用户角色

## 3.1 后台管理员

后台管理员拥有系统管理权限。

主要职责：

* 创建赛事
* 配置报名项目
* 管理会员账号
* 上传 Excel 足环数据
* 查看报名结果
* 审核报名状态
* 导出报名数据
* 维护系统基础数据

## 3.2 会员

会员是实际报名用户。

会员只能访问自己的数据，包括：

* 自己的会员信息
* 自己的足环号码
* 自己的报名记录
* 当前可报名赛事

会员不能查看其他会员的足环号码，也不能提交其他会员的足环号码。

# 4. 核心业务概念

## 4.1 赛事

赛事是报名的最高层级业务对象。

例如：

* 2026 春季大奖赛
* 2026 秋季大奖赛
* 500 公里决赛
* 指定鸽挑战赛
* 特比环决赛

每场赛事下可以配置多个报名项目。

---

## 4.2 报名项目

报名项目是会员可以选择的具体收费项目。

示例：

| 项目名称      | 每注所需羽数 | 每注金额 | 说明        |
| --------- | -----: | ---: | --------- |
| 单羽 50 元   |      1 |   50 | 每只鸽子单独报名  |
| 单羽 100 元  |      1 |  100 | 每只鸽子单独报名  |
| 单羽 200 元  |      1 |  200 | 每只鸽子单独报名  |
| 双羽组 200 元 |      2 |  200 | 2 只鸽子组成一组 |
| 三羽组 300 元 |      3 |  300 | 3 只鸽子组成一组 |
| 五羽组 500 元 |      5 |  500 | 5 只鸽子组成一组 |

系统不能把“单羽、双羽、三羽、五羽”写死，而应该通过核心字段控制：

```text
每注所需羽数 group_size
```

规则如下：

```text
group_size = 1：单羽项目
group_size > 1：多羽组合项目
```

这样管理员未来可以自由创建：

* 单羽组
* 双羽组
* 三羽组
* 五羽组
* 十羽组
* 任意自定义羽数组合

---

## 4.3 足环号码

足环号码是每只赛鸽的唯一识别编号。

Excel 导入时，每个足环号码需要绑定到具体会员。

基础字段：

* 会员棚号
* 参赛名
* 足环号码

会员登录后，只能看到与自己账号绑定的足环号码。

---

## 4.4 报名注

一条报名记录可以理解为一“注”。

### 单羽项目

一注对应 1 只赛鸽。

例如：

```text
单羽 50 元
足环 A 报名一次
金额 = 50 元
```

### 多羽项目

一注对应多只赛鸽组成的一组。

例如：

```text
双羽组 200 元
足环 A + 足环 B 组成一组
金额 = 200 元
```

```text
三羽组 300 元
足环 A + 足环 B + 足环 C 组成一组
金额 = 300 元
```

# 5. 技术栈最终推荐

## 5.1 总体技术路线

本项目推荐采用：

```text
独立 Web 系统
+
移动端 H5 报名页面
+
Laravel 后端 API
+
MySQL 数据库
+
Redis 缓存
+
Nginx 静态资源与反向代理
```

最终推荐：

```text
会员端：Vue 3 + Vite + TypeScript
后台端：Laravel 后台管理面板
后端：Laravel API
数据库：MySQL 8.4 LTS
缓存：Redis
服务器：Nginx + PHP-FPM + OPcache
部署：腾讯云轻量应用服务器
```

---

## 5.2 前端选型

### 5.2.1 会员端前端

推荐：

```text
Vue 3 + Vite + TypeScript + Pinia + Vue Router
```


### 5.2.2 移动端 UI 组件

推荐：

```text
Vant 4
+
自定义报名矩阵组件
```

Vant 适合移动端常规组件，例如：

* 按钮
* 弹窗
* Toast 提示
* Tab 切换
* 搜索框
* 底部固定栏
* 加载状态
* 空状态

但单羽矩阵不建议直接使用普通表格组件，需要自定义。

原因：

1. 单羽矩阵需要固定表头。
2. 单羽矩阵需要固定足环号码列。
3. 项目列需要横向滑动。
4. 每个单元格需要足够大的点击区域。
5. 已选状态需要高度定制。
6. 需要适合手机端而不是传统后台表格。

### 5.2.3 前端状态管理

前端在进入赛事报名页面后，应一次性获取初始化数据，然后在本地维护报名状态。

推荐状态结构：

```text
race：赛事信息
projects：报名项目
pigeons：会员足环列表
singleMatrix：单羽选择矩阵
multiGroups：多羽组合列表
amountSummary：金额汇总
```

前端操作过程中不请求后端。

以下操作都在本地完成：

* 单羽勾选
* 单羽取消
* 多羽选择足环
* 多羽确认成组
* 删除多羽组合
* 实时金额计算
* 已选数量统计
* 报名明细预览

只有以下场景请求后端：

1. 登录。
2. 进入赛事报名页初始化数据。
3. 提交报名。
4. 查看已提交报名。
5. 退出登录。

3. 表单验证和业务校验能力强。
4. 数据库迁移方便。
5. 适合开发中小型业务系统。
6. 适合部署在单台轻量应用服务器上。
7. 业务代码可读性强，后续维护成本较低。
8. 对 Excel 导入导出、后台管理面板有成熟生态。

# 6. 高并发与性能优化方案

## 6.1 并发场景分析

系统峰值访问假设：

```text
300 名会员同时访问赛事报名页面
每个会员 20 - 80 只足环
每场赛事 3 - 15 个报名项目
会员主要操作集中在手机端页面内
提交集中发生在报名截止前
```

这类系统的性能压力主要不是复杂计算，而是：

1. 同时进入报名页导致大量读取赛事配置和足环数据。
2. 会员频繁勾选时如果每次都请求后端，会造成接口压力。
3. 报名提交时如果逐条插入，会造成数据库压力。
4. Excel 导入导出如果和报名高峰同时执行，会影响服务器性能。
5. 静态资源如果不缓存，会浪费带宽和 CPU。

因此系统设计原则是：

```text
初始化时读取一次
前端本地完成交互
提交时一次性写入
后端集中校验
缓存减少重复查询
异步处理耗时任务
```

---

## 6.2 会员端初始化接口

进入赛事报名页面时，前端只调用一个初始化接口。

接口示例：

```http
GET /api/member/races/{race_id}/bootstrap
```

返回数据包括：

```json
{

    "id": 1,
    "name": "2026 春季大奖赛",
    "registration_start_at": "2026-05-01 08:00:00",
    "registration_end_at": "2026-05-01 18:00:00",
    "status": "open",
    "config_version": 12
  },
  "member": {
    "id": 1,
    "loft_number": "A001",
    "participant_name": "张三鸽舍"
  },
  "projects": [
    {
      "id": 1,
      "name": "单羽50",
      "group_size": 1,
      "price_cent": 5000,
      "sort_order": 1
    },
    {
      "id": 2,
      "name": "双羽组200",
      "group_size": 2,
      "price_cent": 20000,
      "sort_order": 2
    }
  ],
  "pigeons": [
    {
      "id": 101,
      "ring_number": "CHN-2026-000001"
    },
    {
      "id": 102,
      "ring_number": "CHN-2026-000002"
    }
  ],
  "existing_registration": null

```

前端拿到这些数据后，在本地完成所有报名操作。

---

## 6.3 前端本地计算原则

以下操作不能频繁请求服务器：

1. 单羽勾选。
2. 单羽取消。
3. 多羽选择足环。
4. 多羽确认成组。
5. 多羽删除组合。
6. 总金额计算。
7. 每个项目已选数量统计。
8. 报名明细生成。
9. 搜索足环。
10. 展开或收起明细。

这些全部由前端本地状态完成。

这样即使 300 人同时操作，服务器也不会因为每一次点击而产生请求。

---

## 6.4 本地草稿保存

为避免会员误刷新或手机切后台导致选择丢失，前端可以使用：

```text
localStorage
或
IndexedDB
```

保存当前报名草稿。

保存内容包括：

* race_id
* member_id
* singleMatrix
* multiGroups
* updated_at
* config_version

注意：

1. 本地草稿只是用户体验优化。
2. 本地草稿不能作为最终可信数据。
3. 提交时后端必须重新校验。
4. 如果赛事项目配置版本发生变化，前端需要提示用户刷新。

---

## 6.5 配置版本号机制

每场赛事需要有一个配置版本号：

```text
config_version
```

当管理员修改以下内容时，版本号递增：

* 赛事状态
* 报名截止时间
* 报名项目名称
* 报名项目金额
* 报名项目 group_size
* 项目启用状态
* 重复使用规则

会员初始化页面时拿到 config_version。

提交报名时，前端带上 config_version。

如果提交时后端发现版本号不一致，则返回：

```text
赛事报名项目已更新，请刷新页面后重新确认报名。
```

这样可以避免会员在旧配置下提交错误报名。

## 6.6 缓存策略

## 6.6.1 赛事配置缓存

缓存 Key：

```text
race:{race_id}:config
```

缓存内容：

* 赛事信息
* 报名项目
* 规则配置
* config_version

缓存时间：

```text
5 - 30 分钟
```

但更推荐：

```text
管理员修改赛事或项目时主动清除缓存
```

这样既能保证性能，也能保证数据准确。

---

## 6.6.2 会员足环缓存

缓存 Key：

```text
member:{member_id}:pigeons
```

缓存内容：

* 当前会员可报名足环列表

缓存时间：

```text
5 - 15 分钟
```

当管理员导入、修改、禁用、删除足环时，清除对应会员足环缓存。

---

## 6.6.3 初始化数据缓存

可以为每个会员和赛事缓存 bootstrap 数据。

缓存 Key：

```text
race:{race_id}:member:{member_id}:bootstrap
```

缓存时间：

```text
3 - 10 分钟
```

适合报名高峰期。

但需要注意：

1. 如果会员已有报名记录发生变化，需要清除缓存。
2. 如果赛事项目更新，需要清除缓存。
3. 如果足环数据更新，需要清除缓存。

---

## 6.6.4 不应缓存的数据

以下数据不能简单缓存：

1. 报名提交结果。
2. 管理员确认状态。
3. 支付状态。
4. 需要实时准确的报名统计。
5. 正在写入的报名事务。

---

## 6.7 提交报名优化

会员提交时，只发起一次请求。

接口：

```http
POST /api/member/races/{race_id}/registrations
```

请求示例：

```json
{
  "config_version": 12,
  "idempotency_key": "uuid-generated-by-client",
  "entries": [
    {
      "project_id": 1,
      "pigeon_ids": [101]
    },
    {
      "project_id": 2,
      "pigeon_ids": [101, 102]
    }
  ]
}
```

后端处理流程：

```text
验证登录身份
↓
验证赛事状态和报名时间
↓
验证 config_version
↓
验证报名项目是否启用
↓
验证 group_size 与 pigeon_ids 数量一致
↓
验证足环属于当前会员
↓
验证重复使用规则
↓
后端重新计算金额
↓
开启数据库事务
↓
写入报名主表
↓
批量写入报名明细
↓
批量写入报名足环关联表
↓
提交事务
↓
清除相关缓存
↓
返回报名结果
```

---

## 6.8 数据库写入优化

报名提交时，不要逐条循环写入数据库。

应使用：

```text
批量 insert
```

例如：

1. 一次写入 registration。
2. 批量写入 registration_entries。
3. 批量写入 registration_entry_pigeons。

避免：

```text
每一只足环单独 insert
每一个项目单独 insert
每一次循环都访问数据库
```

---

## 6.9 幂等提交设计

为避免会员重复点击提交按钮导致重复报名，需要使用幂等机制。

前端生成：

```text
idempotency_key
```

后端建立唯一约束：

```text
member_id + race_id + idempotency_key
```

如果同一个 key 重复提交，后端直接返回第一次提交结果，不重复写入。

同时前端提交按钮进入 loading 状态：

```text
提交中，请勿重复点击
```

---

## 6.10 报名修改策略

MVP 阶段建议：

```text
每个会员每场赛事只有一份有效报名记录
```

如果允许会员在截止前修改，则采用：

```text
读取已有报名
↓
会员修改
↓
提交时替换原报名明细
↓
保留报名主表
↓
更新总金额
```

后端需要在事务中：

1. 锁定该会员该赛事报名主记录。
2. 删除旧明细。
3. 写入新明细。
4. 更新总金额。
5. 记录修改日志。

## 6.11 静态资源优化

前端 Vue 构建后生成静态文件。

Nginx 应设置：

1. JS/CSS 文件长期缓存。
2. 文件名带 hash。
3. 开启 gzip 或 brotli。
4. 图片压缩。
5. 不在报名高峰期重新部署前端资源。

示例策略：

```text
/assets/*.js 缓存 30 天
/assets/*.css 缓存 30 天
index.html 不长期缓存
```

## 6.12 API 优化

API 设计原则：

1. 减少接口数量。
2. 初始化数据一次返回。
3. 避免前端频繁轮询。
4. 避免每次点击都请求后端。
5. 提交时一次性提交完整报名。
6. 后端使用缓存减少重复查询。
7. 后端响应只返回必要字段。
8. 金额统一使用整数分，避免小数误差。

金额字段建议：

```text
price_cent
total_amount_cent
```

例如：

```text
50 元 = 5000 分
```

## 6.14 高峰期服务器策略

报名高峰前：

1. 不进行 Excel 大批量导入。
2. 不进行大数据导出。
3. 不频繁修改赛事项目配置。
4. 预热赛事配置缓存。
5. 预热会员足环缓存。
6. 检查 Redis、MySQL、Nginx、PHP-FPM 状态。
7. 检查磁盘空间。
8. 开启数据库备份。
9. 使用压测工具模拟 300 人访问。

报名高峰中：

1. 禁止后台频繁修改赛事项目。
2. 禁止大批量 Excel 导入。
3. 禁止导出大文件。
4. 管理员统计页面不要自动频繁刷新。
5. 如需查看数据，用轻量统计接口。

报名高峰后：

1. 允许导出完整数据。
2. 允许进行财务统计。
3. 允许生成完整报表。
4. 进行数据库备份。

## 6.15 300 人并发下的理想请求模型

理想情况下，每位会员完整报名只产生少量请求：

```text
1 次登录请求
1 次赛事列表请求
1 次报名初始化请求
1 次报名提交请求
1 次提交结果查看请求
```

其中真正重的请求只有：

```text
报名初始化
报名提交
```

会员页面内所有点击操作都不访问服务器。

如果 300 人同时在线，服务器压力主要是：

```text
300 次初始化请求
+
300 次提交请求
```

这对于经过缓存和批量写入优化后的单机系统是可控的。

# 7. 系统整体业务流程

## 7.1 后台业务流程

### 第一步：创建会员账号

管理员在后台添加会员。

会员字段包括：

| 字段   | 说明         |
| ---- | ---------- |
| 手机号  | 会员登录账号     |
| 密码   | 会员登录密码     |
| 会员棚号 | 会员编号 / 棚号  |
| 参赛名  | 鸽舍名称或参赛人名称 |
| 状态   | 启用 / 禁用    |
| 备注   | 后台备注       |

会员登录方式：

```text
手机号 + 密码
```

会员核心身份字段：

```text
会员棚号 + 参赛名
```

---

### 第二步：上传 Excel 足环数据

管理员上传 Excel 表格。

标准表头：

```text
会员棚号（编号） | 参赛名 | 足环号码
```

系统解析 Excel 后，将足环号码绑定到对应会员。

推荐规则：

1. 会员棚号必须已存在于会员系统。
2. 如果 Excel 中会员棚号不存在，则该行导入失败。
3. 系统生成错误报告，提示管理员处理。
4. 同一足环号码不能重复导入。
5. 足环号码前后空格需要自动清理。
6. Excel 导入前需要有预览确认步骤。

---

### 第三步：创建赛事

管理员创建赛事。

赛事字段包括：

| 字段        | 说明                   |
| --------- | -------------------- |
| 赛事名称      | 例如 2026 春季大奖赛        |
| 赛事说明      | 前端展示的赛事介绍            |
| 报名开始时间    | 会员可开始报名的时间           |
| 报名截止时间    | 报名结束时间               |
| 状态        | 草稿 / 已发布 / 已截止 / 已归档 |
| 是否允许提交后修改 | 控制会员是否可以修改报名         |
| 是否需要后台确认  | 会员提交后是否需要管理员确认       |
| 是否前端显示    | 控制会员端是否可见            |

---

### 第四步：配置赛事报名项目

管理员在赛事下添加报名项目。

项目字段包括：

| 字段         | 说明                  |
| ---------- | ------------------- |
| 项目名称       | 例如单羽 50 元、双羽组 200 元 |
| 每注所需羽数     | 1、2、3、5 等           |
| 每注金额       | 当前项目每注报名金额          |
| 项目说明       | 可选                  |
| 是否启用       | 启用后会员端可选            |
| 排序         | 控制前端显示顺序            |
| 是否允许重复使用足环 | 控制同一足环是否可重复进入同一项目   |
| 每会员最大报名注数  | 可选限制                |
| 每只足环最大报名次数 | 可选限制                |

核心规则：

```text
每注所需羽数 = 1，则前端归入单羽组列表矩阵。
每注所需羽数 > 1，则前端归入多羽组组合报名。
```

---

### 第五步：发布赛事

赛事发布后，会员端才能看到该赛事。

未发布赛事：

```text
会员不可见
```

已发布且在报名时间内：

```text
会员可报名
```

已截止赛事：

```text
会员只能查看，不能继续报名
```

---

### 第六步：后台查看报名结果

管理员可以查看：

* 会员报名明细
* 项目报名统计
* 财务金额统计
* 足环报名统计
* 待确认报名
* 已确认报名
* 已取消报名

后台应支持导出 Excel。

## 7.2 会员端业务流程

### 第一步：会员登录

会员使用手机号和密码登录。

登录成功后显示：

```text
欢迎，张三鸽舍
棚号：A001
```

---

### 第二步：查看可报名赛事

会员登录后进入赛事列表。

显示内容：

* 赛事名称
* 报名截止时间
* 当前状态
* 进入报名按钮

示例：

```text
2026 春季大奖赛
报名截止：2026-05-01 18:00
状态：报名中

[进入报名]
```

---

### 第三步：进入赛事报名页

进入赛事后，页面顶部显示：

* 赛事名称
* 会员棚号
* 参赛名
* 足环数量
* 报名截止时间

示例：

```text
2026 春季大奖赛

会员棚号：A001
参赛名：张三鸽舍
我的赛鸽：48 羽
报名截止：2026-05-01 18:00
```

---

### 第四步：选择单羽组项目

单羽组无论赛鸽数量多少，统一采用“列表矩阵形式”。

也就是：

```text
足环号码作为每一行
单羽项目作为表头列
会员在行列交叉位置点击勾选
```

示例：

```text
足环号码          单羽50    单羽100    单羽200
CHN-2026-001      □         □          □
CHN-2026-002      ✓         □          □
CHN-2026-003      □         ✓          ✓
CHN-2026-004      □         □          ✓
```

会员只需要找到足环号码，然后在对应项目列点击即可。

---

### 第五步：选择多羽组项目

多羽组采用“选择项目 → 选择足环 → 确认成组”的方式。

示例：

```text
选择项目：双羽组 200 元
系统提示：请选择 2 只赛鸽组成一组
```

会员选择：

```text
CHN-2026-001
CHN-2026-005
```

点击：

```text
[确认组成一组]
```

系统生成组合：

```text
双羽组 200 元
第 1 组
CHN-2026-001
CHN-2026-005
金额：¥200
```

三羽组、五羽组逻辑相同，只是要求选择的足环数量不同。

---

### 第六步：实时金额计算

会员每次勾选或取消选择时，系统实时更新金额。

金额组成：

```text
总金额 = 单羽报名金额 + 多羽组合金额
```

底部固定金额栏示例：

```text
单羽 ¥800 ｜ 多羽 ¥400
总计 ¥1,200

[查看明细并提交]
```

---

### 第七步：提交报名

会员点击“查看明细并提交”后进入确认页。

确认页显示：

* 赛事名称
* 会员棚号
* 参赛名
* 单羽报名明细
* 多羽组合明细
* 总金额
* 提交按钮

会员确认无误后点击：

```text
[确认提交报名]
```

---

### 第八步：报名成功

提交成功后显示：

```text
报名提交成功

报名编号：REG202605010001
报名时间：2026-05-01 14:30
总金额：¥1,200
状态：待确认
```

```text
已选 12 项
```

# 9. 单羽组交互设计：固定列表矩阵模式

## 9.1 核心要求

单羽组无论会员有多少只赛鸽，都统一采用列表矩阵形式。

不再使用“每只赛鸽一张卡片”的方式。

核心结构：

```text
横向：单羽报名项目
纵向：足环号码
交叉格：是否报名该项目
```

也就是：

```text
足环号码 × 单羽项目 = 可点击勾选格
```

---

## 9.2 单羽组页面示例

```text
单羽组报名

我的赛鸽：48 羽

搜索足环号码

足环号码          50元      100元      200元
---------------------------------------------
CHN-2026-001      □         □          □
CHN-2026-002      ✓         □          □
CHN-2026-003      □         ✓          ✓
CHN-2026-004      □         □          ✓
```

---

## 9.3 表头固定

单羽项目作为表头，需要在用户上下滚动时保持固定。

原因：

会员可能有几十只足环，滚动到下方后，如果表头不固定，就容易忘记每一列对应哪个项目。

固定表头示例：

```text
足环号码          50元      100元      200元
---------------------------------------------
```

页面上下滚动时，这一行始终显示在顶部。

---

## 9.4 足环号码列固定

如果单羽项目较多，横向空间不足，则右侧项目列可以横向滑动。

但是左侧足环号码列必须固定。

示例：

```text
固定列              可横向滑动项目列

足环号码        |   50元    100元    200元    300元    500元
CHN-001         |    □       □        ✓        □        □
CHN-002         |    ✓       □        □        □        □
CHN-003         |    □       ✓        □        ✓        □
```

要求：

1. 足环号码列始终可见。
2. 项目表头和项目内容横向同步滚动。
3. 用户不能因为横向滑动而看不到当前行的足环号码。

---

## 9.5 行高与点击区域

每一行不能过矮。

推荐：

```text
行高：52px - 60px
单个勾选格点击高度：至少 44px
单个勾选格点击宽度：至少 52px
```

不能只放一个很小的 checkbox。

正确做法：

```text
整个单元格都可以点击
```

未选状态：

```text
□
```

已选状态：

```text
✓
```

更推荐：

```text
未选：浅色边框 + 空心圆
已选：高亮背景 + 对勾
```

---

## 9.6 防止看错行列的 UI 规则

因为单羽矩阵中行列较多，必须重点避免看错、点错。

### 9.6.1 奇偶行弱区分

使用极浅背景区分奇偶行。

示例：

```text
第 1 行：白色
第 2 行：极浅灰
第 3 行：白色
第 4 行：极浅灰
```

这样横向查看时不容易串行。

### 9.6.2 当前操作行高亮

用户点击某个勾选格后，该行短暂高亮。

效果：

```text
刚刚点击的足环号码所在行短暂变亮
```

作用：

1. 让会员知道自己刚刚操作的是哪一行。
2. 避免误以为点到了上一行或下一行。

### 9.6.3 当前操作列高亮

用户点击某个项目列时，该列也可以短暂高亮。

例如点击“单羽 100 元”列时：

```text
单羽100这一列表头和当前格子短暂高亮
```

作用：

1. 让会员确认自己选的是哪个金额。
2. 避免把 50 元、100 元、200 元看错。

### 9.6.4 已选状态必须明显

已选格子不能只靠颜色区分。

必须同时使用：

* 对勾符号
* 背景变化
* 边框变化
* 必要时显示金额简写

示例：

```text
✓ 100
```

这样即使在户外或手机亮度较低时，也能看清是否已选。

## 9.7 足环号码显示规则

足环号码通常较长，手机端容易看花。

建议：

1. 显示完整足环号。
2. 突出显示最后 4–6 位。
3. 字体使用等宽数字或清晰数字字体。
4. 支持复制足环号码。
5. 支持搜索后几位。

示例：

```text
CHN-2026-03-000123
```

视觉重点突出：

```text
000123
```

## 9.8 搜索功能

单羽列表上方必须提供搜索。

搜索支持：

* 完整足环号码
* 足环号码后 3 位
* 足环号码后 4 位
* 足环号码后 6 位
* 连续部分号码

示例：

```text
搜索：123
```

显示包含 123 的足环号码。

搜索要求：

1. 搜索后已选状态不能丢失。
2. 搜索结果中仍然可以继续勾选。
3. 清空搜索后，所有已选状态仍然保留。
4. 搜索框应固定在列表上方，便于快速定位。

## 9.9 单羽项目统计

在单羽列表上方显示每个项目的已选数量和小计金额。

示例：

```text
单羽50：12 羽 / ¥600
单羽100：8 羽 / ¥800
单羽200：3 羽 / ¥600
```

```text
[50元：12羽 ¥600] [100元：8羽 ¥800] [200元：3羽 ¥600]
```

作用：

1. 会员可以快速看到每个项目选了多少只。
2. 方便核对金额。
3. 避免只看总金额不清楚明细。

---

## 9.10 批量操作

批量操作需要谨慎。

MVP 第一阶段可以暂不做全选，避免误操作。

如果后续增加批量操作，建议只提供以下功能：

### 按项目全选当前筛选结果

例如：

```text
将当前筛选出的 48 只赛鸽全部报名【单羽50】
预计增加金额：¥2,400

[取消] [确认全选]
```

### 按项目清空当前筛选结果

例如：

```text
确认清空当前筛选范围内所有【单羽50】报名吗？

[取消] [确认清空]
```

批量操作必须二次确认。

不允许一键直接执行。

## 9.11 单羽组金额计算

每勾选一个交叉格，即产生一条单羽报名。

示例：

```text
CHN-001 勾选 单羽50
CHN-001 勾选 单羽100
CHN-002 勾选 单羽50
```

金额：

```text
50 + 100 + 50 = 200 元
```

# 14. 报名逻辑规则

## 14.1 单羽项目规则

单羽项目定义：

```text
group_size = 1
```

规则：

1. 每个勾选格代表一条报名。
2. 同一足环在同一个单羽项目中只能报名一次。
3. 同一足环可以报名多个不同单羽项目。
4. 取消勾选后报名金额实时减少。
5. 项目被后台禁用后，前端对应列不可选。
6. 报名截止后，所有单羽勾选格不可点击。

---

## 14.2 多羽项目规则

多羽项目定义：

```text
group_size > 1
```

规则：

1. 必须选择刚好等于 group_size 数量的足环。
2. 未选够不能确认成组。
3. 超过数量不能继续选择。
4. 每确认一组，生成一条报名组合。
5. 每组内不能有重复足环。
6. 删除组合后，金额实时减少。
7. 是否允许同一足环进入多个组合，由后台配置控制。

---

## 14.3 金额计算规则

金额组成：

```text
总金额 = 单羽金额 + 多羽金额
```

单羽金额：

```text
单羽金额 = 所有已选单羽格子的项目金额之和
```

多羽金额：

```text
多羽金额 = 所有已确认多羽组合的项目金额之和
```

示例：

```text
CHN-001 报名单羽50 = 50
CHN-002 报名单羽50 = 50
CHN-003 报名单羽100 = 100
CHN-001 + CHN-002 报名双羽组200 = 200
CHN-003 + CHN-004 + CHN-005 报名三羽组300 = 300

总金额 = 50 + 50 + 100 + 200 + 300 = 700 元
```

---

## 14.4 后端金额校验

前端可以实时显示金额，但最终金额必须由后端重新计算。

提交时，前端不应直接传最终金额作为可信数据。

后端需要根据：

* 赛事项目 ID
* 项目价格
* 项目羽数
* 足环数量
* 足环归属
* 报名规则

重新计算最终金额。

如果前端金额和后端金额不一致：

```text
以后端金额为准
```

# 23. 重点难点说明

## 23.1 单羽组的重点是“矩阵清晰度”

单羽组虽然逻辑简单，但会员的足环数量可能很多，项目也可能很多。

如果使用卡片式设计，会导致：

1. 页面过长。
2. 核对困难。
3. 操作效率低。
4. 容易漏选。
5. 容易误选。

因此最终确定：

```text
单羽组一律采用列表矩阵形式。
```

这个设计既适合 5 只足环，也适合 50 只足环。

---

## 23.2 多羽组的重点是“组合逻辑”

多羽组不能用普通矩阵，因为多羽组本质上是“组成一组”。

因此必须使用：

```text
选择项目 → 选择足环 → 确认成组
```

这样会员能清楚知道：

1. 当前报的是哪个项目。
2. 这个项目需要几只鸽子。
3. 当前已经选了几只。
4. 还差几只。
5. 哪些足环组成了一组。
6. 每一组多少钱。

---

## 23.3 后台项目不能写死

系统不能写死：

```text
单羽
双羽
三羽
五羽
```

而必须抽象为：

```text
每注所需羽数 group_size
```

这样未来可以自由配置任意项目。

---

## 23.4 性能重点不是“服务器很强”，而是“请求很少”

本系统最重要的性能设计不是堆服务器配置，而是减少请求数量。

正确设计：

```text
进入页面读取一次
页面内操作全部本地完成
提交时一次性提交
后端一次性校验与写入
```

错误设计：

```text
每次勾选都请求服务器
每次取消都请求服务器
每次金额变化都请求服务器
每次搜索都请求服务器
```

后者会在报名高峰期迅速压垮小型服务器。

---

## 23.5 金额必须保存快照

报名提交时，必须保存：

* 项目名称快照
* 项目金额快照
* 项目羽数快照
* 足环号码快照

原因：

后台后续可能修改项目名称和价格，但历史报名记录不能变化。

---

## 23.6 Excel 导入必须严谨

Excel 数据经常存在：

* 表头不标准
* 足环号码重复
* 足环号码有空格
* 会员棚号不一致
* 空行
* 参赛名写法不一致

所以必须有：

```text
上传 → 预览 → 校验 → 确认 → 导入 → 错误报告
```

不能直接盲目导入。

```text
Vue 3 + Vite + TypeScript + Pinia + Vue Router + Vant + 自定义报名矩阵组件

Laravel API + PHP 8.4 / PHP 8.5

腾讯云轻量应用服务器

```


最终开发原则：

```text
后台高度可配置
前端交互足够简单
单羽报名使用矩阵列表
多羽报名使用组合成组
金额实时前端计算
金额最终后端校验
页面内操作不请求服务器
提交时一次性写入

```

# 27. 最终一句话总结

本系统应该做成一个轻量但结构清晰的独立业务系统：

```text
Vue 负责手机端高效交互
Laravel 负责业务规则和后台管理
MySQL 负责可靠数据存储
Redis 负责缓存和削峰
Nginx 负责静态资源与反向代理
前端本地完成报名选择
后端只在初始化和提交时参与
```

这样才能在小型腾讯云轻量应用服务器上，以较低成本支撑赛鸽赛事报名高峰，同时保证数据准确、交互清晰、后期可维护。
